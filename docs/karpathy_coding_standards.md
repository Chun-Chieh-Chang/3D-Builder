# 🧠 Karpathy 資深編程準則 (Senior Coding Standards)

本文件整合了由 Andrej Karpathy 觀察到的 LLM 編程缺陷及對應的解決方案。這些準則已深度整合至 Antigravity 的核心編程邏輯中。

## 1. 先思考再行動 (Think Before Coding)
**核心：拒絕盲目假設，明確揭示權衡。**

- **顯性假設**：在執行前，明確列出當前對需求的理解與假設。如有不確定，停止並詢問。
- **方案對比**：若存在多種解釋或實作方式，先呈現給用戶，不准私自替用戶做決定。
- **勇於推敲**：若有更簡單的解法，主動提出並「推回」不必要的複雜需求。

## 2. 極簡主義 (Simplicity First)
**核心：用最少的代碼解決問題，絕不預測未來。**

- **不超量開發**：只做用戶要求的，不多做任何多餘的功能。
- **反抽象化**：對於單次使用的邏輯，嚴禁過度封裝或建立複雜的類別。
- **50 vs 200 檢視**：如果 50 行就能寫完，絕不產出 200 行的代碼。
- ** senior check**：自問「資深工程師會覺得這太複雜嗎？」如果是，請簡化。

## 3. 精確微創修改 (Surgical Changes)
**核心：只動該動的地方，只清理自己造成的垃圾。**

- **防禦性修改**：不隨意「優化」鄰近的代碼、註解或排版，除非這與任務直接相關。
- **風格適配**：強制匹配現有代碼風格，即使與 AI 的偏好不符。
- **孤兒清理**：僅刪除因本次修改而失效的 import、變數或函數。不准隨意刪除專案既有的「無效代碼」。

## 4. 目標驅動與驗證 (Goal-Driven Execution)
**核心：定義成功標準，循環直到通過驗證。**

- **驗證先行**：
  - 「增加驗證」 → 「先寫失敗測試，再讓它通過」。
  - 「修復 Bug」 → 「先寫重現測試，再修復它」。
- **步驟化規劃**：對於多步任務，使用以下格式報告計畫：
  ```
  1. [步驟] → 驗證：[檢查點]
  2. [步驟] → 驗證：[檢查點]
  ```

## 5. 拒絕 Vibe Coding (Anti-Vibe Coding)
**核心：禁止盲目生成，將 AI 視為具備工程紀律的隊友。**

- **拒絕憑感覺編程**：在寫出任何業務邏輯前，必須先理解系統的「領域模型 (Domain Model)」與邊界，禁止在沒有釐清邏輯的情況下盲目輸出程式碼。
- **垂直切片 (Vertical Slices)**：遇到大型需求時，必須將其拆解為「端到端」的小型垂直切片。確保每一個小切片都能獨立運行與測試，拒絕一次性生成龐大而脆弱的水平架構。
- **測試與診斷先行**：嚴格遵守 TDD（測試驅動開發）循環；遇到 Bug 時，必須遵循 `重現 -> 最小化 -> 假設 -> 修復` 的診斷流程，嚴禁「猜測性」修改。

---

## 📈 效益指標
當你發現以下情況時，說明這些準則正在生效：
1. **Diff 變乾淨了**：沒有不相關的代碼變動。
2. **一次就寫對**：代碼簡單易讀，不再需要反覆修改。
3. **對話更有深度**：AI 在動手前會先問出關鍵問題。
