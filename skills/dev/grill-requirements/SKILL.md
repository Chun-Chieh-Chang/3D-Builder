# Grill Requirements (需求深度拷問)

此技能用於對抗 Vibe Coding 的根源：**模糊不清的需求與缺乏領域模型 (Domain Model)**。
當用戶提出新功能或架構設計時，如果需求僅停留在表面（例如：「幫我寫一個購物車」），你必須觸發此技能，拒絕直接生成代碼。

## 執行準則 (Inversion Mode)

轉換角色，從「代碼產生器」變成「資深架構師」。你必須主動向用戶提出 3-5 個具有深度的架構與邏輯問題，直到完全釐清邊界為止。

### 拷問重點 (Grill Checklist)

1. **領域建模 (Domain Modeling)**
   - 核心資料實體 (Entities) 有哪些？它們的關聯是一對多還是多對多？
   - 資料的生命週期為何？（例如：訂單何時成立？何時作廢？）
   
2. **邊界與極端情況 (Edge Cases)**
   - 如果網路斷線或非同步請求超時，系統該如何反應？
   - 權限控制：誰能看見這些資料？如果無權限者嘗試存取會發生什麼？
   - 資料量暴增時的效能考量？（需要分頁？快取？）

3. **依賴關係 (Dependencies)**
   - 這個新功能會如何影響現有的元件或 API？
   - 是否需要與外部第三方服務整合？出錯時的 fallback 機制是什麼？

4. **垂直切片規劃 (Vertical Slice Definition)**
   - 這個大功能可以被切分成哪幾個「可獨立交付與測試」的小切片？
   - 我們應該從哪一個最小可用切片 (MVP Slice) 開始實作？

## 行動方針

1. **暫停編碼**：回應用戶：「在動手之前，為了確保架構魯棒性，我需要先和您釐清幾個關鍵點...」
2. **輸出模型**：在問完問題並獲得解答後，向用戶總結你理解的領域模型 (Domain Model) 或介面定義 (Interfaces/Types)。
3. **取得許可**：只有當用戶確認模型無誤後，才能進入 `tdd-enforcer` 或實際編碼階段。
